1. Executive Summary
The Decentralized KYC (Know Your Customer) System is a blockchain-based identity verification platform that enables a user to verify their identity once and reuse that verified identity across an unlimited number of institutions and entity types ù without ever transmitting their original document to any party.
Version 3.0 expands the system from a two-node model (Node 1 + one additional bank) to a true Multi-Entity Ecosystem. The platform now natively supports any combination of entity types operating as verifying nodes, including:
Banks and financial institutions (retail, investment, neobanks)
Cryptocurrency exchanges (CEX and DEX)
Insurance companies (life, health, general)
Government portals (tax authorities, passport offices, e-government services)
Healthcare providers (hospitals, clinics, pharmacies)
DeFi and Web3 platforms (lending protocols, yield aggregators, DEXs)
Telecom operators (mobile and broadband providers)
Any other regulated or compliance-driven entity type
The system's core innovation remains its Privacy-Preserving Multi-Node Hash Verification Protocol. When a user uploads their identity document to the KYC DApp, a SHA-256 cryptographic hash is generated entirely within the user's browser. The document itself never leaves the user's device. Only this 64-character hash fingerprint is stored on the Ethereum blockchain.
When any entity node ù regardless of type ù needs to verify the same user, the user uploads their document to the DApp again. The DApp regenerates the hash client-side and sends only the hash to the entity's API endpoint. The entity node then cross-checks this hash against the immutable blockchain record, granting or denying access based on whether the hashes match. The original document is never seen by any entity node at any point.
A new on-chain Entity Registry allows the admin to register, configure, and manage any number of entity nodes across all supported entity types. Each entity is identified by type, has configurable permissions, and can be added or removed dynamically without redeploying the contract.
This design is fully GDPR-compliant, fraud-resistant, and production-grade ù closely mirroring how enterprise identity systems like Sovrin and KILT Protocol operate in the real world, while extending their reach to the full breadth of the regulated economy.
2. Problem Statement
2.1 Current KYC Challenges
Redundancy: Users submit identity documents to every bank, exchange, insurer, government portal, or platform separately
Document Exposure: Each new institution receives, stores, and processes a full copy of sensitive documents ù multiplying breach risk with every new submission
No User Control: Users have no visibility or control over who holds copies of their passport, ID, or utility bills
High Cost: Financial institutions globally spend over $1.8 billion annually on KYC compliance
Slow Onboarding: Manual KYC can take 3û14 business days per institution
Centralized Data Silos: Each institution's KYC database is a honeypot for hackers
Sector Fragmentation: Banks, insurers, exchanges, and government agencies each run completely separate KYC processes ù even for the same user
2.2 The Core Privacy Problem This System Solves
Even in systems that reuse KYC across institutions, the standard approach still requires the user to send their document to every subsequent institution. This means the document travels across networks and gets stored in multiple places ù at every bank, every insurer, every exchange, every government portal.
This system solves that entirely and at scale. The document is hashed on the user's own device. No institution beyond Node 1 ever receives or stores the document ù not the bank, not the crypto exchange, not the insurance company, not the government portal. Every entity node, regardless of type, only ever receives a 64-character mathematical fingerprint which is cryptographically useless without the original document.
2.3 The Multi-Entity Gap This Version Addresses
Version 2 demonstrated that a second node (a single extra bank) could verify a user via hash comparison. However, real-world identity reuse spans sectors, not just banks. A user who is KYC-verified at their primary bank should be able to instantly onboard to a crypto exchange, apply for insurance, access government services, and register on a DeFi protocol ù all without ever submitting documents again.
Version 3 closes this gap by introducing a universal multi-entity architecture that treats all institutional types as first-class verifying nodes within a single unified system.

5. Functional Requirements
5.1 Client-Side Hashing Module
FR-01: Document Upload & Hash Generation
When a user selects their identity document on the DApp, the frontend immediately computes a SHA-256 hash of the file using the browser's built-in Web Crypto API. The document itself is never uploaded to any server or transmitted anywhere ù regardless of which entity type the user is interacting with.
Input: User selects file via file picker (passport, ID, utility bill, etc.)
Processing: SHA-256 hash computed entirely inside the browser using Web Crypto API
Output: 64-character hexadecimal hash string displayed to the user
Constraint: File never leaves the user's device at any point in this step
Performance: Hash generation completes in under 1 second for files up to 10MB
5.2 Smart Contract Module
FR-02: User Registration
Any Ethereum wallet address can register by calling registerUser() with their name and the SHA-256 hash generated client-side. This creates a User record on-chain.
Input: name (string), documentHash (string ù SHA-256 hex)
Output: UserRegistered event emitted on blockchain
Constraint: Each wallet address can only register once
FR-03: Node 1 Verification with Hash Confirmation
Node 1 (the primary verifier) approves the user after reviewing their document off-chain. During approval, Node 1 also submits the confirmed document hash ù which must match what the user registered. This locks the verified hash permanently on-chain.
Input: userAddress (address), confirmedHash (string)
Smart contract checks: keccak256(registeredHash) == keccak256(confirmedHash)
If match: isVerified = true, verifiedHash locked, verifiedBy = Node1 address
If mismatch: Transaction reverted with 'Hash mismatch ù document not matching'
Output: UserVerified event emitted with verifier address and timestamp
FR-04: Multi-Entity Hash Verification
This is the core innovation of v3. When any registered entity node ù a bank, crypto exchange, insurance company, government portal, healthcare provider, DeFi platform, telecom operator, or any other type ù needs to verify a user, the user uploads their document on the DApp. The DApp generates the hash client-side and sends only the hash to the entity's API endpoint. The entity calls checkDocumentHash() on the smart contract to cross-check.
Input: userAddress (address), submittedHash (string from DApp), entityType (enum)
Smart contract checks: submittedHash == verifiedHash AND isVerified == true AND entityType is registered
If all true: HashCheckPassed event emitted with entity type and address, returns true
If any false: HashCheckFailed event emitted with entity type and address, returns false
Critical: No entity node ever receives or stores the original document
Critical: The document hash is generated on the user's device ù not on any entity's server
Critical: The same verifiedHash is used for all entity types ù one hash, universal reuse
FR-05: KYC Revocation
Any authorized verifier can revoke a user's KYC status at any time ù for fraud, expired documents, or compliance violations. Revocation propagates instantly to all entity types.
Input: userAddress (address)
Output: isVerified = false, UserRevoked event emitted permanently on-chain
Effect: All subsequent hash checks for this user return false immediately ù across all entity types without exception
FR-06: Entity Registry Management
The admin manages the on-chain Entity Registry, which tracks all authorized verifying nodes across all entity types.
registerEntity(address, entityType, name, apiEndpoint): Admin registers a new entity node with its type classification
removeEntity(address): Admin removes an entity from the registry
updateEntityEndpoint(address, newEndpoint): Admin or entity updates API endpoint
getEntityInfo(address): Returns entity type, name, API endpoint, and status for any address
Entity types enum: BANK, CRYPTO_EXCHANGE, INSURANCE, GOVERNMENT, HEALTHCARE, DEFI, TELECOM, OTHER
Constraint: Only admin (contract deployer) can register or remove entities
FR-07: Verifier Management
addVerifier(address): Admin grants node verifier privileges (Node 1 capability)
removeVerifier(address): Admin revokes verifier privileges
Constraint: Only admin (contract deployer) can manage verifiers

7. Technical Architecture & Stack
7.1 System Architecture
The system follows a three-tier architecture with no traditional backend server. The smart contract is the backend ù all business logic, entity registry, access control, and data storage happen on-chain. The only off-chain communication is when the DApp sends a hash (not a document) to an entity node's API endpoint for cross-checking.
In v3, the architecture scales horizontally across entity types. Each entity type (bank, exchange, insurer, etc.) operates its own independent API endpoint. All entity endpoints follow the same protocol: receive {walletAddress, documentHash}, call checkDocumentHash() on the smart contract, return the result. The smart contract is the single source of truth for all entity types simultaneously.

8. System Flow & User Journeys
8.1 First-Time Verification Flow (Node 1)
Step 1 ù User connects MetaMask wallet to the KYC DApp
Step 2 ù User enters their name and selects their identity document (passport, ID card, etc.)
Step 3 ù DApp instantly computes SHA-256 hash of the document using Web Crypto API in the browser. Document never moves. Hash is displayed to the user for transparency.
Step 4 ù User calls registerUser(name, hash) via MetaMask ù transaction confirmed on blockchain
Step 5 ù Node 1 (bank or KYC provider) reviews the actual document off-chain via email or their existing portal
Step 6 ù Node 1 computes the same hash from the document they reviewed, then calls verifyUser(userAddress, confirmedHash) on the smart contract
Step 7 ù Smart contract verifies: confirmedHash must match the user's registered hash. If match, isVerified = true and verifiedHash is locked permanently on-chain. If mismatch, transaction is reverted.
Step 8 ù UserVerified event emitted. User is now KYC-verified on blockchain and eligible for verification by any registered entity node.
8.2 Multi-Entity Access Request Flow (Any Entity Type)
Step 1 ù User wants access to an entity (e.g., a crypto exchange, an insurer, a government portal, a hospital, a DeFi protocol)
Step 2 ù User visits the DApp and selects the entity type (e.g., CRYPTO_EXCHANGE) and the specific institution (e.g., Binance)
Step 3 ù User uploads their identity document on the DApp again
Step 4 ù DApp generates SHA-256 hash of the document client-side in the browser. Document is not sent anywhere.
Step 5 ù DApp sends only the hash + user's wallet address + entity type to the selected entity's registered API endpoint
Step 6 ù Entity's API calls checkDocumentHash(walletAddress, receivedHash, entityType) on the smart contract
Step 7 ù Smart contract checks: does receivedHash match the verifiedHash on-chain? Is isVerified true? Is the calling entity registered?
Step 8 ù If all true: HashCheckPassed event emitted with entity type and address; entity grants access. If any false: HashCheckFailed event emitted; entity denies access.
Step 9 ù The entity never receives, stores, or processes the original document at any point.
Step 10 ù The same flow works identically for Banks, Exchanges, Insurance companies, Government portals, Healthcare providers, DeFi platforms, Telecom operators, and any other registered entity type.
8.3 Fraud Detection via Hash Mismatch
If a user attempts to submit a tampered or different document to any entity node, the SHA-256 hash generated will be different from the verified hash stored on blockchain. Even changing a single pixel of an image produces a completely different hash. The mismatch is detected instantly and a HashCheckFailed event ù including the entity type ù is permanently recorded on-chain. This mechanism is universal across all entity types.
8.4 Revocation Flow
Step 1 ù A verifier detects fraud, expired documents, or a compliance violation
Step 2 ù Verifier calls revokeUser(userAddress) on the smart contract
Step 3 ù isVerified is set to false. UserRevoked event is permanently emitted on-chain.
Step 4 ù All subsequent hash checks for this address return false immediately ù across all entity types, all nodes, all sectors. Revocation is universal.
9. Security Design
9.1 Privacy by Design ù The Hash Protocol
Document hashing occurs exclusively client-side using the browser's native Web Crypto API
The original document is never transmitted to any server, entity node, or blockchain
All entity nodes ù regardless of type ù only ever receive a 64-character hex string
A SHA-256 hash cannot be reverse-engineered back into the original document
Fully compliant with GDPR Article 25 (Data Protection by Design and by Default)
9.2 Fraud Prevention via Hash Immutability
The verifiedHash is locked on-chain by Node 1 at the time of verification
No one ù not even the admin ù can modify a stored hash after it is written
Any document alteration (even 1 pixel) produces a completely different SHA-256 hash
Hash mismatches are permanently recorded as HashCheckFailed events with entity type on-chain
The on-chain audit trail is immutable ù fraud attempts cannot be erased and are visible to all entities
9.3 Smart Contract Access Control
onlyAdmin modifier: Only contract deployer can add or remove verifiers and manage the entity registry
onlyVerifier modifier: Only whitelisted node addresses can verify, confirm hashes, or revoke
onlyRegisteredEntity modifier: Only addresses in the entity registry can call checkDocumentHash()
Hash match validation: verifyUser() reverts if Node 1's hash doesn't match user's registered hash
Entity type validation: checkDocumentHash() verifies the calling entity's type matches its registry entry
No Ether transfers in contract ù eliminates reentrancy attack surface
Solidity ^0.8.0 ù built-in overflow and underflow protection
